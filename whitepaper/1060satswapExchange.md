# 10.6 SatSwap Exchange Message Spec

## The Exchange Primitive

SatSwap defines a single, minimal exchange between two nodes: one node requests a block and pays for it; the other delivers the block and receives payment. This exchange is the primitive operation of the entire protocol.

The handshake does not distinguish between roles. A node requesting a block may be a client requesting content for end-user delivery, or an intermediate node requesting a block it does not hold locally in order to serve a downstream request. A node delivering a block may be a storage host, a caching node, or any intermediate node that holds the block and is willing to serve it at a stated price. The exchange is identical in all cases.

This role-agnosticism is load-bearing. It means the same implementation handles every hop in a multi-node swarm delivery without modification. The swarm behavior described in Section 3.6 is not a separate mode — it is the natural result of the same four-message handshake executing in parallel across multiple node pairs simultaneously.

---

## 10.6.1 Design Principles

**Minimal.** The spec defines only what is necessary to complete an atomic exchange. Everything beyond the four messages — discovery, reputation, routing strategy, block assembly — is handled by other layers.

**Atomic.** The requesting node does not pay unless it receives a verifiable block. The delivering node does not release the block unless payment is confirmed. Neither party can be cheated by the other through the normal operation of the protocol.

**Implementation-agnostic.** The message definitions specify fields and semantics. Wire format, serialization, and transport are left to the implementer. A conforming implementation may use Protocol Buffers, CBOR, JSON, or any other encoding, provided the required fields are present and correctly interpreted.

**Stateless per exchange.** Each four-message handshake is self-contained. No persistent session state is required between node pairs. Nodes may exchange blocks with any number of counterparties simultaneously without coordination.

---

## 10.6.2 The Four-Message Handshake

Every SatSwap exchange follows this sequence without exception:

```
Requesting Node          Delivering Node
      |                        |
      |-------- WANT --------->|
      |                        |
      |<------- QUOTE ---------|
      |                        |
      |----- PAYMENT_HASH ---->|
      |                        |
      |<------- BLOCK ---------|
      |                        |
```

The exchange completes when the requesting node receives a BLOCK message containing data that verifies against the requested CID. If any step fails or times out, the exchange is abandoned and the requesting node may retry with a different counterparty.

---

## 10.6.3 Message Definitions

### WANT

Sent by the requesting node to initiate an exchange. States what block is being requested and the maximum price the requesting node is willing to pay.

```
WANT {
  cid:          string    // The content identifier of the requested block
  max_msats:    uint64    // Maximum price the requester will pay, in millisatoshis
  request_id:   string    // Unique identifier for this exchange instance
}
```

**Field notes:**

- `cid` identifies the specific block being requested. For a multi-block file, each block has its own CID within the Merkle DAG. The requesting node issues one WANT per block.
- `max_msats` communicates the requester's price ceiling. The delivering node uses this to decide whether to respond with a QUOTE or ignore the request.
- `request_id` is generated by the requesting node and must be unique per exchange. It is echoed in subsequent messages to correlate the handshake.

---

### QUOTE

Sent by the delivering node in response to a WANT. States the price the delivering node will accept and provides a Lightning invoice for that amount.

```
QUOTE {
  request_id:   string    // Echoed from the WANT message
  cid:          string    // Echoed from the WANT message
  msats:        uint64    // Price the delivering node will accept, in millisatoshis
  invoice:      string    // BOLT11 Lightning invoice for the stated amount
  expires_at:   uint64    // Unix timestamp after which this quote is no longer valid
}
```

**Field notes:**

- `msats` must be less than or equal to `max_msats` from the WANT. A delivering node that cannot serve the block at or below the requester's price ceiling should not respond.
- `invoice` is a standard BOLT11 Lightning invoice. The payment hash embedded in the invoice is the commitment that binds the payment to this specific exchange.
- `expires_at` gives the requesting node a deadline for payment. Quotes that are not paid before expiry are abandoned by the delivering node.

---

### PAYMENT_HASH

Sent by the requesting node after paying the Lightning invoice. Provides proof of payment initiation and the payment hash that the delivering node will use to verify settlement before releasing the block.

```
PAYMENT_HASH {
  request_id:     string    // Echoed from prior messages
  cid:            string    // Echoed from prior messages
  payment_hash:   string    // Hash of the Lightning payment preimage
}
```

**Field notes:**

- `payment_hash` is the hash of the HTLC preimage. It proves that a payment has been initiated and committed. The delivering node verifies this against the invoice it issued before proceeding.
- The delivering node does not release the block until it has verified that the payment hash corresponds to the invoice from its QUOTE and that the payment has been received or is in-flight via the Lightning Network.

---

### BLOCK

Sent by the delivering node after payment is confirmed. Contains the raw block data.

```
BLOCK {
  request_id:     string    // Echoed from prior messages
  cid:            string    // The CID of the delivered block
  data:           bytes     // Raw block data
  preimage:       string    // Lightning payment preimage, proving final settlement
}
```

**Field notes:**

- `data` is the raw bytes of the block. The requesting node verifies that hashing `data` produces `cid`. If the hash does not match, the block is invalid and the exchange has failed — the requesting node has paid for a block it cannot use, which is a violation detectable and attributable to the delivering node via the payment record.
- `preimage` is the Lightning payment preimage, which serves as proof of final settlement. Its presence confirms that the delivering node has received payment and considers the exchange complete.

---

## 10.6.4 Failure Handling

The four-message handshake has defined failure points at each step.

**No QUOTE received.** The delivering node does not hold the block, cannot meet the price ceiling, or is offline. The requesting node waits for a configurable timeout, then retries with a different counterparty from the discovery results.

**QUOTE received but price exceeds max_msats.** This should not occur if the delivering node correctly evaluates the WANT — a well-behaved delivering node does not issue a QUOTE above the requester's stated ceiling. If it does, the requesting node discards the QUOTE and treats the exchange as failed.

**Quote expires before payment.** The requesting node failed to pay within the window specified by `expires_at`. The delivering node abandons the exchange. The requesting node may issue a new WANT to restart.

**Block data fails CID verification.** The requesting node receives a BLOCK but the hash of `data` does not match `cid`. This indicates a corrupted or fraudulent delivery. The requesting node retries with a different counterparty. The failed delivery is attributable to the delivering node via the payment record and may be reported as a negative reputation signal to the discovery layer.

**Payment in-flight but no BLOCK received.** The most sensitive failure case. The payment has been initiated but the delivering node has not responded with a BLOCK. The HTLC timeout mechanism of the Lightning Network protects the requesting node — if the preimage is not revealed within the HTLC timelock, the payment is returned. The requesting node should wait for HTLC resolution before retrying.

---

## 10.6.5 The Handshake in Swarm Context

When the Host Discovery Layer returns multiple hosts for a CID, the requesting node initiates parallel handshakes — one per host per block. Each handshake is independent. A WANT sent to Host A has no relationship to a WANT sent to Host B, even if both are requesting blocks for the same file.

In a swarm delivery, intermediate nodes execute the same handshake in both directions simultaneously: receiving a WANT from a downstream node and issuing a WANT to an upstream node for a block they do not hold locally. The intermediate node's margin — the spread between what it charges downstream and what it pays upstream — is determined by the difference between the `msats` fields in the two exchanges.

The protocol has no visibility into whether a given node is an endpoint or an intermediate hop. Nor does it need to. The atomicity of each individual handshake guarantees that every node in the chain is paid for what it delivers, and no node pays for what it does not receive.

---

## Summary

The SatSwap exchange is a four-message handshake — WANT, QUOTE, PAYMENT_HASH, BLOCK — between any two nodes. It is the protocol primitive from which all delivery topologies are composed. It is role-agnostic, stateless per exchange, and implementation-agnostic in wire format. Every hop in a swarm delivery, every single-host direct delivery, and every intermediate proxy operation uses this same handshake without modification. The atomicity of the exchange is enforced by the Lightning Network's HTLC mechanism: the delivering node cannot receive payment without releasing the preimage, and the requesting node cannot be charged for a block it does not receive.
