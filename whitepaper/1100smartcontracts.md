# 11. Smart Contract Operations ⚙️

The IPFS-Sats smart contract is the operational expression of Key 3 — the Execution Key in the three-key architecture (Section 4.1). Key 3 is the automated layer that carries out governance decisions, distributes yield, pays hosts, and manages the LYW without requiring human action on every transaction. All contract logic is driven by the parameters defined in the DAO Configuration Object (Section 10.3) and operates within the authority boundaries established by Key 1 and Key 2.

Key 3 does not make governance decisions — it executes them. Decisions are made by Key 2 (the human authority layer) through the proposal and voting process. Key 3's role is to translate verified votes into state changes and to automate the recurring economic operations that would otherwise require continuous manual intervention. This separation of decision authority from execution authority is the architectural guarantee that automation serves the DAO's members rather than capturing it.

---

## 11.1 Yield Distribution

Key 3 manages the automated, non-custodial disbursement of all revenue generated by the content — access fees, zaps, and inbound fork royalties — adhering to the Fork Economics model established in Section 8.

### Revenue Flow

All incoming revenue follows a two-step flow:

**Step 1 — Upstream royalty out.** If the content is a derivative work (`is_fork: true`), Key 3 immediately routes the `upstream_percentage` defined in the `fork_provenance` fields (Section 10.4) to the Parent content's LYW address. This happens atomically upon receipt — the upstream royalty is never held in the Child LYW.

**Step 2 — Remainder into the Child LYW.** The remaining revenue — `child_dao_percentage` of the total — flows into the Child LYW. From there, the `yield_split` defined in the DAO Configuration Object (Section 10.3) governs how much is distributed to DAO members in the current cycle versus compounded back into the LYW's deployed balance. The compound portion increases the capital available for Lightning liquidity yield generation, strengthening the content's long-term persistence.

There is no separate "persistence fee" bucket. The LYW is the DAO's wallet. The compound/distribute ratio is the persistence mechanism. Content that compounds more grows its economic runway; content that distributes more pays its members sooner. Both are valid choices governed by the DAO.

### Payment Triggers

Revenue events that trigger Key 3 distribution logic include:

- **On-demand access payment** — direct Lightning payment upon content retrieval
- **Zap receipt** — supporter micropayment routed to the LYW address
- **Fork royalty inflow** — upstream royalty payment received from a Child DAO
- **Subscription settlement** — periodic payment for accrued usage under a subscription licensing arrangement
- **Escrow release** — payment upon verification of licensing conditions for a commercial arrangement

### Distribution Reporting

All distribution events are recorded in the LYW State Ledger (Section 10.5) and linked via the `distribution_history_cid` field. This provides a verifiable, auditable history of all economic activity for the DAO — queryable by any participant through the Records Database.

## 11.2 Host Payments

Key 3 pays hosts for content delivery directly through the AtomicSats exchange mechanism. There is no separate proof-of-storage verification system.

### AtomicSats as Proof of Availability

The AtomicSats four-message handshake (Section 10.6) is itself the proof of availability. A host that delivers a valid block in response to a WANT — a block whose hash verifies against the requested CID — has proven it holds that block at that moment. Payment is atomic with delivery: the HTLC settles when the preimage is revealed, simultaneously confirming delivery and releasing payment. No separate challenge-response mechanism is needed because the exchange primitive already provides the necessary proof.

### Payment Flow

Host payments flow from the LYW as AtomicSats exchanges complete. Key 3 monitors completed exchanges and settles host payments per the pricing terms advertised in the host's Host Registry Record. The LYW State Ledger tracks these outflows in the `host_payments_sats` field of the current cycle.

### Drawdown Mode

When the LYW balance drops below the hosting cost threshold, Key 3 automatically transitions to drawdown mode (`drawdown_mode: true` in the LYW State Ledger). In drawdown mode, host AtomicSats payments are suspended. The remaining LYW balance continues to draw down for creator distribution only, ensuring the creator continues to receive yield while the content's economic viability is evaluated. No debt is incurred — expenses are never committed beyond what the LYW can currently pay.

### Recovery from Drawdown

When new revenue arrives and the LYW balance recovers above the hosting cost threshold, Key 3 automatically exits drawdown mode:

1. `drawdown_mode` flag flips to false
2. Host AtomicSats payments resume at the normal rate
3. The `cycles_at_threshold` counter in the LYW State Ledger resets to zero, pausing the sunset countdown
4. If the recovered balance is sufficient to redeploy capital into Lightning liquidity positions, Key 3 re-engages liquidity leasing to restart passive yield generation

Recovery is automatic and requires no DAO vote. It is a Key 3 operational function, not a governance event.

## 11.3 DAO Governance Execution

Key 3 serves as the executive engine for the Per-Content DAO, translating verified votes into state changes according to the governance rules defined in Section 10.3 and the proposal types defined in Section 6.2.

### Proposal Lifecycle

Key 3 enforces the proposal lifecycle defined in the DAO Configuration Object:

1. **Submission** — a member with income share above `proposal_threshold` submits a proposal referencing a specific `allowed_actions` function and proposed parameters
2. **Voting period** — open for the duration defined by `voting_period_blocks`, measured in Bitcoin blocks
3. **Verification** — Key 3 checks that the vote meets both `approval_quorum` (participation threshold) and the required passage threshold for the proposal type (simple majority for operational proposals, supermajority for constitutional proposals — Section 6.2)
4. **Execution delay** — if the vote passes, Key 3 waits `execution_delay_blocks` before executing, giving all participants time to review the outcome
5. **Execution** — Key 3 executes the designated function from the `allowed_actions` list against the mutable state

### Voting Mechanism

Votes are weighted by each member's current income share percentage, as defined in the DAO Configuration Object's `initial_members` array and updated by any subsequent membership governance actions. Members cast votes by cryptographically signing a message referencing the proposal's hash. Key 3 verifies signatures against the current member list before counting votes.

### State Execution

When a vote passes and the execution delay has elapsed, Key 3 executes the approved action and creates a new content-addressed state snapshot — a new `mutable_state_cid` — reflecting the updated operational parameters. The prior state is not deleted; the chain of state CIDs provides a complete, auditable governance history. The new `mutable_state_cid` is written to the DAO Configuration Object and becomes the authoritative operational state for all subsequent Key 3 actions.

### Key 1 Constraints on Execution

Key 3 cannot execute any action that conflicts with the Key 1 configuration established at publication time. If Key 1 is locked, Key 3 will not execute distribution reallocation or membership changes regardless of vote outcome — those actions are structurally unavailable. If Key 1 is configured as multi-sig, actions requiring Key 1 authority require the configured Key 1 signers to co-sign before Key 3 proceeds. Key 3 is the executor; Key 1 is the constitutional boundary.

## 11.4 Lifecycle Management

Key 3 monitors the content's economic health continuously and manages the lifecycle transitions defined in the DAO Configuration Object.

### Sunset Evaluation

Key 3 increments the `cycles_at_threshold` counter in the LYW State Ledger each distribution cycle that the total LYW balance remains below `sunset_threshold_sats`. When `cycles_at_threshold` reaches the duration configured in the `sunset_condition`, Key 3 initiates the sunset process.

Sunset evaluation is suspended whenever `cycles_at_threshold` resets — either because the balance recovers above the threshold or because a DAO vote explicitly resets the counter. A DAO that wants to extend its runway despite a low balance can vote to adjust the sunset threshold or inject new capital into the LYW, either of which resets the evaluation.

### Sunset Execution

When the sunset condition is confirmed, Key 3 executes the `archival_action` defined in the DAO Configuration Object. For the common `public_domain_release` configuration, Key 3 updates the content's license CID to the CC0 public domain dedication, publishes an updated Metadata Wrapper, and anchors the final state to Bitcoin. The content becomes freely accessible, and the DAO's active economic operations cease.

### Key Replacement

If Key 3 itself is compromised or becomes unresponsive, Key 1 and Key 2 cooperate to replace it — Key 2 provides the human authority vote and Key 1 provides the content binding that proves continuity of ownership. This fail-safe ensures that no single key compromise can permanently disable the DAO's governance capability. The replacement process is itself an `allowed_actions` entry (`replace_key`) requiring supermajority vote plus Key 1 co-signature.

---

## Summary

Key 3 automates the four core operational functions of the Per-Content DAO: yield distribution, host payments, governance execution, and lifecycle management. It operates within the authority boundaries set by Key 1 (constitutional constraints) and Key 2 (human governance decisions). Revenue flows into the LYW, upstream royalties route out atomically, and the remainder is governed by the compound/distribute ratio the DAO configures. Hosts are paid through AtomicSats exchange completions — no separate proof-of-storage mechanism is required. Governance decisions made by Key 2 are executed by Key 3 after the voting period, verification, and execution delay defined in the DAO Configuration Object. Lifecycle transitions — drawdown mode, recovery, and sunset — are automated Key 3 responses to the content's economic health as tracked in the LYW State Ledger. Key 3 executes; it does not decide.
